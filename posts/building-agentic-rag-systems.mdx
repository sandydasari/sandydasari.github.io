Most RAG demos retrieve once, stuff context into a prompt, and call it done. Production systems need more: routing across indexes, reflecting on retrieved chunks, and falling back gracefully when relevance is weak.

## The shape of a real RAG pipeline

A useful agentic RAG loop tends to converge on four stages:

1. **Plan** — decompose the user query into sub-questions.
2. **Route** — pick the right index, tool, or API per sub-question.
3. **Retrieve & reflect** — score chunk relevance, retry with rewritten queries if signal is weak.
4. **Synthesize** — cite sources, return structured output.

The reflection step is what separates a toy from a tool. A single round of "did this actually answer the question?" cuts hallucinations dramatically.

## What I'd skip

You don't need a heavy agent framework on day one. A small state machine, well-defined tools, and a decent reranker outperform most agent SDKs for narrow domains. Add framework abstractions only when you have orchestration pain to solve.

## What to measure

- Answer faithfulness (claims supported by retrieved context)
- Retrieval precision @ k
- End-to-end latency p95
- Cost per successful answer

Optimize the last one last. Reliability comes first.
